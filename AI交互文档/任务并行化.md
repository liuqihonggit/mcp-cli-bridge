# /spec 模式并行执行规范

> 📌 **适用场景**: 触发 `/spec` 命令进行规格化开发时
> ⚠️ **目标**: 最大化并行效率，避免文件冲突和依赖问题
> 🔗 **与渐进式开发的关系**: 本规范适用于单次修改涉及多个文件分区操作的场景；小型任务请遵循 CLAUDE.md 的渐进式开发方法（每次只完成一个功能）

## 🎯 核心原则

**触发 `/spec` 时必须使用并行智能体处理多个文件，通过任务编号明确依赖关系**

## 📝 任务编号规则

### 大写字母编号（A, B, C...）- 可并行的独立任务

**特征**:
- 任务之间无依赖关系
- 操作不同的文件/模块
- 可以同时执行而不冲突
- 执行顺序不影响结果

**示例**:
```markdown
## 并行任务列表

A. 创建 IUserService 接口（文件: Contracts/IUserService.cs）
B. 创建 UserService 类（文件: Common/Services/UserService.cs）
C. 创建 UserController 类（文件: McpHost/Controllers/UserController.cs）
D. 编写单元测试（文件: tests/UserServiceTests.cs）

✅ 以上任务可以4个智能体同时执行
```

### 小写字母编号（a, b, c...）- 必须串行的依赖任务

**特征**:
- 依赖前置任务的输出
- 需要等待某个文件创建完成
- 涉及同一文件的多次修改
- 有严格的执行顺序要求

**示例**:
```markdown
## 串行任务列表

a. 等待任务A完成后：在 McpHost/Program.cs 中注册 IUserService (依赖 A)
b. 等待任务B完成后：编写 UserService 的集成测试 (依赖 B)
c. 等待任务A+B完成后：更新 README.md 架构图 (依赖 A + B)

⚠️ 以上任务必须按顺序执行
```

## 📋 完整示例

### 场景：实现用户认证功能

```markdown
# Spec: 用户认证模块实现

## 第一阶段：可并行执行（大写）

A. 创建 IAuthService 接口
   - 文件: src/Common.Contracts/Interfaces/IAuthService.cs
   - 定义: LoginAsync, RegisterAsync, ValidateTokenAsync

B. 创建 AuthService 实现
   - 文件: src/Common/Services/AuthService.cs
   - 实现 IAuthService 接口

C. 创建 AuthController 控制器
   - 文件: src/McpHost/Controllers/AuthController.cs
   - 暴露 MCP 工具: auth_login, auth_register

D. 创建认证相关的 DTO
   - 文件: src/Common.Contracts/DTOs/LoginRequest.cs
   - 文件: src/Common.Contracts/DTOs/RegisterRequest.cs
   - 文件: src/Common.Contracts/DTOs/AuthResponse.cs

E. 创建单元测试框架
   - 文件: tests/AuthServiceTests.cs
   - 准备 Mock 数据和测试用例

## 第二阶段：串行依赖（小写）

a. [依赖 A+B] 在 Program.cs 注册服务
   - 文件: src/McpHost/Program.cs
   - 添加: builder.Services.AddScoped<IAuthService, AuthService>()

b. [依赖 C] 配置路由和中间件
   - 文件: src/McpHost/Program.cs
   - 添加: app.MapControllers()

c. [依赖 D+E] 完善测试用例
   - 文件: tests/AuthServiceTests.cs
   - 使用真实 DTO 进行完整测试

d. [依赖全部] 更新文档
   - 文件: README.md
   - 添加认证模块架构说明
```

## ✅ 并行执行检查清单

### 任务分解阶段

- [ ] 明确所有需要创建/修改的文件列表
- [ ] 分析文件之间的依赖关系
- [ ] 标识可并行的独立任务（大写）
- [ ] 标识有依赖的串行任务（小写）
- [ ] 验证无循环依赖

### 执行阶段

- [ ] 先启动所有大写任务（并行）
- [ ] 监控各任务执行状态
- [ ] 大写任务完成后，按顺序启动小写任务
- [ ] 每个任务完成后立即编译验证
- [ ] 记录每个任务的执行结果

### 冲突避免规则

#### ❌ 禁止的并行操作

1. **同一文件的多次修改**
   ```markdown
   ❌ 错误:
   A. 修改 Program.cs (添加服务注册)
   B. 修改 Program.cs (配置中间件)
   
   ✅ 正确:
   A. 修改 Program.cs (添加服务注册)
   b. [依赖 A] 修改 Program.cs (配置中间件)
   ```

2. **有隐式依赖的任务**
   ```markdown
   ❌ 错误:
   A. 创建 IUserService 接口
   B. 创建 UserService 实现 (假设已存在接口)
   
   ✅ 正确:
   A. 创建 IUserService 接口
   b. [依赖 A] 创建 UserService 实现 (明确依赖)
   ```

3. **可能产生合并冲突的操作**
   ```markdown
   ❌ 错误:
   A. 修改 GlobalUsings.cs (添加命名空间X)
   B. 修改 GlobalUsings.cs (添加命名空间Y)
   
   ✅ 正确:
   A. 修改 GlobalUsings.cs (一次性添加所有命名空间)
   ```

#### ✅ 推荐的并行操作

1. **不同层的代码生成**
   ```
   Contracts层 (接口/DTO) ← → Common层 (实现) ← → McpHost层 (控制器)
   ```

2. **独立的工具/功能模块**
   ```
   Auth模块 ← → User模块 ← → Log模块
   ```

3. **测试与实现分离**
   ```
   测试代码编写 ← → 实现代码编写 (最后集成)
   ```

## 🔧 工具使用建议

### 并行启动多个智能体

```python
# 伪代码示例
tasks = [
    Task(agent="code-expert", task="实现任务A", files=["IAuthService.cs"]),
    Task(agent="code-expert", task="实现任务B", files=["AuthService.cs"]),
    Task(agent="test-writer", task="实现任务E", files=["AuthServiceTests.cs"]),
]

# 并行执行
execute_parallel(tasks)

# 等待所有大写任务完成
wait_for_completion(tasks)

# 串行执行小写任务
execute_serial([task_a, task_b, task_c, task_d])
```

### 依赖检测自动化

```markdown
## 自动化检查项

1. 文件依赖图生成
   - 输入: 任务列表 + 文件列表
   - 输出: 依赖关系矩阵
   
2. 循环依赖检测
   - 如果发现 A→B→A 的依赖，报错
   
3. 并行度优化
   - 建议最大并行数: CPU核心数 / 2
   - 避免过多并行导致资源竞争
```

## 📊 效率对比

### 传统串行方式

```
时间轴: [任务A][任务B][任务C][任务D][任务E][任务a][任务b][任务c][任务d]
总耗时: 9个时间单位
```

### 并行优化方式

```
时间轴: 
┌─[任务A]─┐
├─[任务B]─┤
├─[任务C]─┤ 同一时间窗口
├─[任务D]─┤
└─[任务E]─┘
         [任务a][任务b][任务c][任务d]
总耗时: 5个时间单位 (节省44%)
```

## 🚨 常见错误

| 错误 | 原因 | 解决方案 |
|-----|------|---------|
| 未识别隐式依赖 | 假设接口已存在 | 明确列出所有前置条件 |
| 过度并行导致冲突 | 贪多求快 | 保守估计依赖关系 |
| 小写任务顺序错误 | 忽略传递依赖 | 绘制完整依赖图 |
| 缺少编译验证点 | 盲目信任并行结果 | 每个任务后都编译 |
| 文件锁冲突 | 多个进程写同一文件 | 确保操作不同文件 |

## 💡 最佳实践

### 1. 渐进式复杂度提升

```markdown
第一次使用 /spec 时：
- 只分2-3个大写任务
- 串行任务不超过3个
- 重点掌握流程

熟练后：
- 大写任务可达5-8个
- 复杂依赖链可达10+个任务
- 引入自动化依赖分析
```

### 2. 失败处理策略

> ⚠️ 与 CLAUDE.md 错误分级策略保持一致，禁止遇到小错误就立即回滚

```markdown
如果某并行任务失败：

1. 先评估错误级别（参考 CLAUDE.md 错误分级）
   - ⚪ 小错误（语法/拼写/缺少引用）→ 直接修复，不回滚
   - 🟡 中等问题（接口不匹配/类型冲突）→ 调整方案，不回滚
   - 🔴 严重问题（架构冲突/AOT不兼容）→ 回滚+切分任务
2. 小/中等问题：在当前代码基础上修复，最多尝试2次
3. 严重问题：停止其他并行任务，回滚失败任务，降级为串行执行
4. 手动修复后重新执行
```
