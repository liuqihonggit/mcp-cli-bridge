# 文件锁设计文档

## 1. 设计目标

在多智能体并发访问文件的场景下，确保**任何文件读写操作都必须持有锁**，防止读到半写入文件或产生写冲突。

核心约束：

- **跨进程安全**：不同进程的智能体同时操作同一文件时互斥
- **编译时强制**：LLM 生成的代码若直接使用 `System.IO` API 会编译失败
- **死锁预防**：批量文件操作按路径排序获取，逆序释放（区间锁模型）
- **NativeAOT 兼容**：不依赖反射、动态类型、JSON 直接解析

## 4. 区间锁模型（获取正序，释放逆序）

```
输入: ["C:\z.json", "A:\a.json", "C:\b.json"]
    │
    ├─ 获取阶段（正序）
    │   Path.GetFullPath + Distinct + OrderBy → ["A:\a.json", "C:\b.json", "C:\z.json"]
    │   │
    │   ├─ Lock(A:\a.json)  ✅
    │   ├─ Lock(C:\b.json)  ✅
    │   └─ Lock(C:\z.json)  ✅
    │
    │   ⚠️ 任一步失败 → ReleaseAll(completed) → TimeoutResult/ErrorResult
    │   ℹ️  ReleaseAll 在失败回滚时也是逆序释放
    │
    └─ 释放阶段（逆序）
        DisposeAsync() / ReleaseAll():
        │
        ├─ Unlock(C:\z.json)  ← 最后获取的最先释放
        ├─ Unlock(C:\b.json)
        └─ Unlock(A:\a.json)
```

### 实现细节

```
// 获取: 正序遍历 sorted 列表
// 每文件双层锁: SemaphoreSlim（进程内）→ Mutex（跨进程）
foreach (var filePath in sorted)
{
    // Layer 1: 进程内互斥（SemaphoreSlim）
    var sem = s_intraLocks.GetOrAdd(mutexName, _ => new SemaphoreSlim(1, 1));
    sem.WaitAsync(remaining);

    // Layer 2: 跨进程互斥（Named Mutex）
    mutex = new Mutex(false, mutexName);
    mutex.WaitOne(remaining);
    WriteLockFile(filePath);
    acquiredMutexes.Add((filePath, mutex));
}

// 释放: 逆序遍历（先获取的后释放）
// ReleaseAll() —— 用于获取失败回滚
for (var i = acquired.Count - 1; i >= 0; i--)
{
    acquired[i].Mutex.ReleaseMutex();   // 先释放 OS 锁
    acquired[i].Mutex.Dispose();
    acquiredSem[i].Semaphore.Release();  // 再释放进程内锁
}

// HybridBatchLock.DisposeAsync() —— 正常释放
for (var i = _acquired.Count - 1; i >= 0; i--)
{
    var (path, mutex) = _acquiredMutexes[i];
    DeleteLockFile(path, _provider);  // 先清理调试文件
    mutex.ReleaseMutex();             // 再释放 OS 锁
    mutex.Dispose();                  // 最后销毁句柄
}
for (var i = _acquiredSemaphores.Count - 1; i >= 0; i--)
{
    _acquiredSemaphores[i].Semaphore.Release();  // 释放进程内锁
}
```

### 关键设计点

| 步骤 | 作用 |
|------|------|
| **获取排序** | 所有进程按相同字典序获取，消除循环等待（死锁预防） |
| **释放逆序** | 区间锁标准模型：后获取的先释放，保持加锁区间的对称性 |
| **去重** | `Distinct(StringComparer.OrdinalIgnoreCase)`，同一文件不重复加锁 |
| **全量失败回滚** | 获取任一文件失败 → 已获取的逆序释放 → 返回失败 |
| **remaining timeout** | 总超时 = `指定的timeout`，每步消耗扣减 |

## 5. 死锁预防

### 5.1 问题场景

```
进程A: Lock(B.json) → Lock(A.json)
进程B: Lock(A.json) → Lock(B.json)
         ↑ 循环等待 → 死锁
```

### 5.2 解决方案：全局排序

```csharp
// AcquireBatchAsync 内部:
var sorted = filePaths
    .OrderBy(p => p, StringComparer.OrdinalIgnoreCase)
    .ToList();
```

所有进程都按 `A.json → B.json` 的顺序获取，不存在交叉：

```
进程A: Lock(A.json) → Lock(B.json)    ← 字典序
进程B: Lock(A.json) → Lock(B.json)    ← 相同顺序，无死锁
```

### 5.3 逆序释放的意义

获取 `A → B → C`，释放 `C → B → A`。双层释放顺序：

| 步骤 | 操作 | 原因 |
|------|------|------|
| 1 | `DeleteLockFile(C)` | 先清理调试文件 |
| 2 | `Mutex.Release(C)` | 释放跨进程 OS 锁 |
| 3 | `Semaphore.Release(C)` | 释放进程内锁 |

虽然 Mutex 和 Semaphore 本身不依赖释放顺序，但逆序释放保证了：
- 最后获取的资源最先归还，减少后续等待者的排队时间
- 释放跨进程锁（Mutex）优先于进程内锁（Semaphore），让其他进程尽早获取
- 如果未来引入资源分配图追踪，逆序释放天然避免交叉边

### 5.4 单文件操作

单文件读写（`ReadJsonAsync` 等）内部调用 `AcquireBatchLockAsync([filePath])`，即 N=1 的批量锁。路径与多文件一致，无死锁风险。

## 11. 设计决策记录

| 决策 | 选择 | 备选方案 | 原因 |
|------|------|---------|------|
| 锁类型 | **混合锁（Hybrid）** | 纯 Named Mutex / 纯 SemaphoreSlim / 文件锁 | Semaphore 处理进程内线程互斥 + Mutex 处理跨进程互斥，详见第12节 |
| API 模式 | 纯批量锁 | 单锁 + 批量锁 | 统一代码路径，消除接口歧义 |
| 获取排序 | 字典序 `OrdinalIgnoreCase` | 不排序 | 确定性排序，消除死锁 |
| 释放顺序 | **逆序** | 正序 / 无序 | 区间锁标准模型：后获取的先释放；跨进程锁优先释放 |
| 双层获取序 | SemaphoreSlim → Mutex | Mutex → Semaphore | 先占进程内计数器，再争 OS 内核对象，避免无效 OS 竞争 |
| 双层释放序 | Mutex → SemaphoreSlim | Semaphore → Mutex | 先释放 OS 锁让其他进程尽早获取，再释放进程内锁 |
| 异步包装 | `Task.Run(() => mutex.WaitOne())` | 同步阻塞 | Mutex 获取微秒级，线程池开销可接受 |
| 运行时检测 | ❌ 已移除 | 运行时抛异常 | 编译时 Analyzer 已足够，避免线上崩溃 |
| Analyzer 范围 | `System.IO.File` + 5 种类型 | 所有 System.IO | 够用且不误报 |

## 12. Named Mutex 原语陷阱：为什么不能单独用 Mutex

> 💡 **这是本项目的核心经验教训。**

### 12.1 问题场景

最初的设计只使用 **Named Mutex** 作为唯一锁原语。在跨进程场景下表现正确，但在**同进程多线程**场景下出现了致命缺陷。

```csharp
// ❌ 纯 Named Mutex 方案（已被废弃的 LocalFileLockProvider）
var mutex = new Mutex(false, mutexName);
mutex.WaitOne(timeout);  // 获取锁
// ... 文件操作 ...
mutex.ReleaseMutex();    // 释放锁
```

测试 `ConcurrentAccess_SameProcessTwoThreads_SecondShouldTimeout`:
- 线程1 获取 Mutex → 成功
- 线程2 尝试获取**同一个** Mutex，期望超时 → **结果：直接成功！**

### 12.2 根本原因：Mutex 的线程递归特性

Named Mutex 是 **OS 内核对象**，其所有权模型基于 **线程 ID**：

| 行为 | 说明 |
|------|------|
| **线程递归** | 同一个 OS 线程可以**无阻塞地多次获取**同一个 Mutex |
| **所有权计数** | Mutex 内部维护 `(ThreadID, RecursionCount)` |
| **同线程不阻塞** | 如果当前 OS 线程已持有 Mutex，再次 `WaitOne()` 直接返回 |

这意味着 Mutex 的"互斥"是对**不同线程**而言的，而不是对**不同调用**而言的。

### 12.3 线程池放大了问题

.NET 的 `Task.Run` 使用**线程池**调度任务。线程池会**复用线程**：

```
线程池线程 Thread-1 执行 TaskA → 获取 Mutex → 执行完毕 → 归还线程池
线程池线程 Thread-1 执行 TaskB → 尝试获取 Mutex → 直接成功！
                                      ↑ 因为 Thread-1 还"持有"这个 Mutex
                                      （取决于 ReleaseMutex 是否已被调用）
```

更微妙的情况——**并发竞争**：

```
TaskA 在 Thread-1 上获取 Mutex → 持有中
TaskB 被调度到 Thread-1（线程池复用） → WaitOne() 直接成功！
     ↑ Mutex 认为 Thread-1 已经有权限了
```

这就导致：**两个本应互斥的并发操作，在同一个进程内可能同时执行**。

### 12.4 解决方案：混合锁（HybridFileLockProvider）

引入 **SemaphoreSlim(1,1)** 作为进程内锁层，配合 Named Mutex 作为跨进程锁层：

```
┌──────────────────────────────────┐
│  获取流程（正序）                  │
│                                  │
│  SemaphoreSlim.WaitAsync()       │  ← Layer 1: 进程内互斥
│       ↓ (计数器递减 1→0)         │     基于"计数器"而非"线程ID"
│  Mutex.WaitOne()                 │  ← Layer 2: 跨进程互斥
│                                  │     基于"线程ID"（跨进程有效）
├──────────────────────────────────┤
│  释放流程（逆序）                  │
│                                  │
│  Mutex.ReleaseMutex()            │  ← 先释放 OS 锁，让其他进程进入
│  SemaphoreSlim.Release()         │  ← 再释放进程内锁
│       ↓ (计数器递增 0→1)         │
└──────────────────────────────────┘
```

**关键设计点**：

| 设计点 | 说明 |
|--------|------|
| `SemaphoreSlim(1, 1)` | 最大计数=1，初始=1，退化为二值信号量（等效于进程内 Mutex） |
| 基于**计数器**而非线程ID | `WaitAsync()` 递减计数器到 0，任何线程再调都会阻塞，不会递归 |
| `static ConcurrentDictionary` | 同进程内对同一文件路径复用同一个 SemaphoreSlim 实例 |
| 获取先 Semaphore 后 Mutex | 先在进程内排队，避免所有线程都去争 OS 内核对象（减少系统调用） |
| 释放先 Mutex 后 Semaphore | 先释放跨进程锁让其他进程尽早获取，最后释放进程内锁 |

### 12.5 对比总结

| 特性 | 纯 Named Mutex | 混合锁 (Hybrid) |
|------|---------------|----------------|
| 跨进程互斥 | ✅ | ✅ |
| 同进程多线程互斥 | ❌ (线程递归破坏) | ✅ (SemaphoreSlim 基于计数器) |
| 死锁预防 | ✅ (排序获取) | ✅ (排序获取 + 逆序释放) |
| 性能 | 好 (纯 OS 内核对象) | 略增 (多一层 SemaphoreSlim) |
| 复杂度 | 低 | 中 |

### 12.6 教训

> **Named Mutex 的"互斥"是线程级别的，不是调用级别的。**
>
> 如果需要在**同一进程的不同线程间互斥**，必须使用基于**计数器**而非**线程 ID** 的同步原语（SemaphoreSlim、Semaphore、Monitor 等）。
>
> Named Mutex 的线程递归特性是**设计特性而非 bug**——它能防止单线程死锁自己。但在用 `Task.Run` 并发调度任务的场景下，这个特性恰恰变成了互斥的破坏者。
