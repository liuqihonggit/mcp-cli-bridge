# 渐进式无人执行模式 - 完整操作手册

> 📎 **定位**: 本文档详细定义任务执行的渐进式处理机制
> 🔗 **引用**: 在 [CLAUDE.md](../CLAUDE.md) 中作为"环境特殊要求"的扩展
> 🎯 **目标**: 80%的任务应在 Level 1-2 内解决，Level 3 是安全网而非常态

## 核心理念

**尽可能自主完成，遇到问题渐进式升级处理方式，而非立即放弃或交互**

---

## 📋 阶段1：任务规划与边界定义（开始前必做）

### 1. 构建目录树

- 开始任何任务前，必须先列出相关目录结构
- 明确涉及的文件范围和影响区域
- 记录当前 git 状态作为基线

### 2. 任务分解与打勾机制

```
✅ 任务1: [具体任务描述]
⬜ 任务2: [具体任务描述]
⬜ 任务3: [具体任务描述]
...
```

- 每个任务必须有明确的**成功标准**（如：编译通过、测试通过）
- **完成一个任务后立即打勾 ✅ + 立即 git commit，再进行下一个**
- 禁止同时处理多个未完成任务
- 禁止累积多个打勾任务后再统一提交

### 3. 边界范围控制

- 明确任务的**输入/输出**
- 定义**不做的事项**（防止范围蔓延）
- 预估风险点和回滚预案

---

## 🔄 阶段2：渐进式修改流程（执行中遵循）

### � 核心总则（不可违反）

> **每完成一个任务，必须立即 git commit。编译成功 ≠ 完成，编译+提交 = 完成。**

```
任务完成标准 = 代码修改 + 编译通过 + git提交
```

- ✅ 即使是修改单个文件的小任务，完成后也必须提交
- ✅ 禁止"等几个任务一起提交"或"最后一起提交"
- ✅ 提交信息应简洁描述本次修改内容

---

### �� 安全前置条件（神圣不可侵犯）

```powershell
# 步骤1: 保证 Git 环境干净
git --no-pager status
git --no-pager diff

# 步骤2: 创建安全备份点
git add -A
git commit -m "backup: [任务名称] 开始前的安全点"

# 步骤3: 验证备份成功
git --no-pager log -1
```

### ✅ 成功里程碑设计

| 里程碑 | 触发条件 | 动作 |
|--------|----------|------|
| M1: 编译 src 通过 | `dotnet build src` 无错误 | git commit |
| M2: 编译 tests 通过 | `dotnet build tests` 无错误 | git commit |
| M3: 单元测试通过 | `dotnet test` 全绿（TDD 红绿循环产出） | git commit |
| M4: E2E 验证完成 | E2E 测试通过（模拟真实客户端/服务端交互） | git commit |
| M5: 发布验证完成 | `.\build.ps1` 构建成功（AOT 发布产物正确） | git commit |

> ⚠️ **M3 说明**: 单元测试必须通过 TDD 红绿重构循环产出，禁止先写生产代码后补测试
> ⚠️ **M4 说明**: 每个对外接口必须有 E2E 测试覆盖协议边界，防止参数顺序/格式隐性回归
> ⚠️ **审查规则**: E2E 测试变更必须人工逐行确认，单元测试变更可快速过审

> ⚠️ **强制**: 每达到一个里程碑必须立即 git 提交，禁止累积多个里程碑再提交

### 🔴 TDD 红绿重构循环（写代码时强制）

> **铁律**: 没有失败测试，就不允许写生产代码

```
每个功能的开发循环:
  🔴 红: 写一个会失败的测试 → 编译确认测试失败
  🟢 绿: 写最小代码让测试通过 → 编译确认测试通过
  🔵 重构: 优化代码结构 → 确认测试仍然通过
  → git commit → 下一个功能
```

**顺序要求**:
- 先写测试验证**行为**（做什么），后补测试验证**输入**（边界条件）
- ⛔ **禁止补测试**: 发现生产代码没有测试时，禁止直接补测试
  - 正确: 先移除生产代码（移到 `.x/`），再从写失败测试开始
  - 原因: 补测试会依赖旧代码结构，破坏 TDD 驱动的正确设计
- **例外**: 纯 DTO/枚举等无逻辑代码无需 TDD

### 📁 文件操作规范（神圣不可改变）

#### 1. ⛔ 禁止删除文件（不可协商）

> **违反此规则 = 任务立即失败 = 不可接受的行为**

**🚫 绝对禁用：**

| 禁用项 | 类型 | 说明 |
|--------|------|------|
| `DeleteFile` 工具 | Trae IDE 内置工具 | **永远不要调用此工具** |
| `Remove-Item` | PowerShell 命令 | 禁止使用 |
| `del` / `rm` | 命令行命令 | 禁止使用 |
| 任何删除操作 | 所有形式 | 无例外 |

**为什么这是红线？**
- 删除 = 永久丢失 = 无法回滚
- 破坏渐进式安全原则
- 丢失审计追踪能力

```powershell
# ❌ 绝对错误：任何删除操作
Remove-Item oldfile.cs
del oldfile.cs
# 调用 DeleteFile 工具
```

#### 2. ✅ 唯一正确做法：移动到 .x/ 目录

```powershell
# ✅ 正确：安全移动并标记
if (-not (Test-Path ".x")) { New-Item -ItemType Directory -Path ".x" }
$timestamp = Get-Date -Format "yyyyMMdd_HHmmss"
Move-Item "oldfile.cs" ".x\oldfile.cs.$timestamp.del"
```

- **格式**: `.x/{文件名}.{原后缀}.{时间戳}.del`
- 如果 `.x/` 目录不存在则自动创建
- 确保 `.gitignore` 排除 `.x/` 目录

#### 3. ✅ 文件迁移流程

```powershell
# 步骤1: 先用 PowerShell 移动到新目录
Move-Item "src/Old/MyClass.cs" "src/New/MyClass.cs"

# 步骤2: 移动完成后才修改命名空间
# （使用 SearchReplace 工具，不用 PowerShell）
```

### ⚙️ 子智能体协作规范

| 场景 | 主智能体 | 子智能体 | 备注 |
|------|----------|----------|------|
| 修改命名空间 | 规划+监督 | 执行批量替换 | 推荐 |
| 修复少量错误 (<10个) | 自行修复 | 可选派发 | 灵活选择 |
| 文件内容修改 | **禁止用 PowerShell** | **禁止用 PowerShell** | 只能用 SearchReplace |
| 大规模重构 (>50个文件) | 先 git 备份 | 可用脚本批量处理 | 必须有备份点 |

### ⚠️ 乱码检测与处理

当检测到文件乱码时：

1. **先评估**: 判断乱码范围（单个文件 vs 多个文件）
2. **单文件乱码**: 回滚该文件的最近修改
3. **多文件乱码**: 
   - 评估是否需要 git 回滚到上一个里程碑
   - 检查是否是编码问题（UTF-8 BOM 等）
4. **记录决策**: 在工作文件末尾记录乱码原因和处理方式

## 🚨 阶段3：分级错误处理与降级策略（遇到错误时）

> **预防针**: 遇到错误不要惊慌！按照以下级别逐步升级处理

### 🟢 Level 1: 小错误（<30 个编译错误/警告）

**特征**: 语法错误、拼写错误、缺少引用、命名空间问题等

**处理方式**:

- ✅ **自行修复**: 直接在当前代码基础上修改
- ✅ **派发子智能体**: 如果是重复性错误（如批量改命名空间），可派子智能体并行处理
- ✅ **重试次数**: 最多尝试 2 次不同方案
- ❌ **不回滚**: 保持当前进度，基于已有修改继续优化

**示例**:

```
检测到 15 个编译错误:
  - 12个: 命名空间未更新 → 派子智能体批量替换
  - 3个: 缺少 using 引用 → 自行添加
→ 修复后重新编译 → 成功 → 提交 git
```

---

### 🟡 Level 2: 中等问题（30-100 个报错）

**特征**: 接口不匹配、类型冲突、架构调整导致的连锁错误、大量依赖断裂等

**处理方式**:

#### 1. 评估阶段 (必须做)

- 分析错误类型分布（是同类错误还是多种错误混合？）
- 判断根因（是规划问题还是执行偏差？）
- 评估修复成本 vs 回滚成本

#### 2. 决策分支

**选项A: 调整方案继续执行** (推荐优先尝试)

- 细化任务粒度，拆分为更小的子任务
- 逐个子任务修复并验证
- 每完成一个子任务就编译验证
- **⚠️ 降级拆分粒度**: 拆到单文件单函数级别，确保每步可独立编译
- **⛔ 禁止整体回滚再重来**: 看到大量错误就回滚会陷入"回滚→重试→再回滚"死循环
- 正确做法: 在当前进度上，将剩余错误拆成最小可修复单元逐个击破

**选项B: Git 回滚重试** (如果评估显示方向错误)

```powershell
git --no-pager checkout -- .
git --no-pager clean -fd
```

- 回滚到上一个安全备份点
- 重新规划任务，采用不同方案
- 从头开始执行

#### 3. 重试限制

- 总共 **3 次**机会（含自行修复和回滚重试）
- 每次必须使用**不同的方案**
- 记录每次失败的原因和尝试的方案

#### 4. 触发降级条件

- 连续 2 次修复失败且错误数量未明显下降
- 第 3 次尝试仍然失败 → 进入 Level 3 用户接管模式

**决策记录格式**:

```markdown
<!-- 🔄 Error Handling: Level 2 -->
<!-- 时间: 2026-04-30 -->
<!-- 错误数: 67 个 -->
<!-- 根因分析: [具体原因] -->
<!-- 决策: [选项A/B] -->
<!-- 方案详情: [如何修复或如何重新规划] -->
<!-- 结果: [成功/失败] -->
```

---

### 🔴 Level 3: 大型错误（>100 个报错）或 重试用尽

**特征**: 架构冲突、NativeAOT 不兼容、大规模代码损坏、方向性错误等

**处理方式**:

#### 0. 穷尽手段检查（触发 Level 3 交互前必须逐项确认）

> **原则**: 只有穷尽所有自主手段后，才允许请求用户交互

- [ ] 🔍 已查询 MCP 工具（MemoryCli 记忆）— 无相关经验
- [ ] 📂 已搜索项目代码（SearchCodebase/Grep）— 无可参考实现
- [ ] 🛠️ 已检查可用技能（Skill 工具）— 无相关能力
- [ ] 🌐 已搜索互联网（WebSearch/WebFetch）— 无有效方案
- [ ] 以上全部确认仍无法解决 → 允许进入用户交互

#### 1. ❌ 立即停止自动执行

- 禁止继续尝试自行修复
- 保护当前状态，避免进一步破坏

#### 2. 📊 生成诊断报告

```markdown
<!-- 🛑 Task Failed: [任务名称] -->
<!-- 尝试次数: X/3 -->
<!-- 错误数量: XXX 个 -->
<!-- 错误类型分布: -->
<!--   - 类型A: XX个 (示例) -->
<!--   - 类型B: XX个 (示例) -->
<!-- 已尝试方案: -->
<!--   - 方案1: [描述] → 结果: [失败原因] -->
<!--   - 方案2: [描述] → 结果: [失败原因] -->
<!--   - 方案3: [描述] → 结果: [失败原因] -->
<!-- 当前Git状态: [clean/dirty] -->
<!-- 最后成功点: [commit hash or message] -->
<!-- 需要用户决策: -->
<!--   A. 继续尝试其他方案 -->
<!--   B. 回滚到某个历史版本 -->
<!--   C. 改变任务目标/范围 -->
<!--   D. 手动介入修复 -->
```

#### 3. 💬 触发用户交互对话

- 使用 `AskUserQuestion` 工具（此时允许交互）
- 向用户展示诊断报告
- 提供清晰的选项供用户决策
- 等待用户指令后再继续

## 🔄 阶段4：回滚后的重启机制（回滚触发后必做）

> **核心原则**: 回滚不是结束，而是重新开始的起点

### 重启检查清单

```markdown
## 🔄 回滚后重启清单

### 1. 环境验证
- [ ] Git 状态干净 (`git --no-pager status`)
- [ ] 代码已恢复到预期版本 (`git --no-pager log -3`)
- [ ] 编译环境正常 (`dotnet build src` 能成功)

### 2. 问题复盘
- [ ] 分析上次失败的根本原因
- [ ] 记录到经验记忆（避免重蹈覆辙）
- [ ] 识别是否有遗留修改未清理

### 3. 遗留检查（关键步骤）
- [ ] 检查 `.x/` 目录是否有意外移动的文件
- [ ] 检查是否有未提交的临时文件
- [ ] 检查命名空间是否已完全还原
- [ ] 检查 GlobalUsings.cs 是否被误改

### 4. 重新规划
- [ ] 基于复盘结果调整任务方案
- [ ] 细化任务粒度（比上次更细）
- [ ] 设置更多的中间里程碑
- [ ] 准备更频繁的 git 提交点

### 5. 重新执行
- [ ] 从第一个任务重新开始
- [ ] 每步都严格验证
- [ ] 发现异常立即停止，不再强行推进
```

### 防遗漏机制

- 回滚后必须从**第一个任务**重新开始，不能跳过
- 每完成一个任务都要对比上次的执行记录
- 特别关注上次失败的节点，提前设置检查点
- 如果连续 2 次在同一位置失败，必须暂停并请求用户指导

## 📈 执行模式总结图

```
┌─────────────────────────────────────────────────────┐
│  开始任务                                           │
│  ↓                                                  │
│  🔍 阶段1: 规划 + 边界定义 + 目录树                 │
│  ↓                                                  │
│  🔒 创建 Git 安全备份点                              │
│  ↓                                                  │
│  🔄 阶段2: 渐进式执行（TDD 循环）                   │
│  ├─ 🔴红: 写失败测试                                │
│  ├─ 🟢绿: 写最小代码通过                            │
│  ├─ 🔵重构: 优化代码                                │
│  └─ → 编译 → git提交 → 下一个功能                   │
│  ↓                                                  │
│  🚨 遇到错误？                                      │
│  ↓                                                  │
│  ├─ <30个 → 🟢 Level1: 自行修复/派子智能体          │
│  │   └─ 成功 → 继续下一步                           │
│  │   └─ 失败(2次) → 进入Level2                      │
│  ↓                                                  │
│  ├─ 30~100个 → 🟡 Level2: 评估+降级拆分/回滚(3次)   │
│  │   ├─ 选项A(优先): 拆到单文件单函数级别逐个击破    │
│  │   ├─ ⛔ 禁止: 整体回滚再重来(死循环)             │
│  │   └─ 选项B: 仅方向错误时才回滚                    │
│  ↓                                                  │
│  └─ >100个 → 🔴 Level3: 停止 + 报告 + 用户决策     │
│      └─ 用户决定: 继续/回滚/改变目标/手动介入        │
│                                                     │
│  💾 回滚后?                                         │
│  └→ 阶段4: 重启检查清单 → 从任务1重新开始           │
└─────────────────────────────────────────────────────┘
```

### 错误分级速查

| 级别 | 错误数量 | 处理方式 | 重试限制 | 是否交互 |
|------|----------|----------|----------|----------|
| 🟢 L1 | <30 | 自行修复 / 派子智能体 | 2次 | ❌ |
| 🟡 L2 | 30-100 | 评估后调整或回滚 | 3次(总) | ❌ |
| 🔴 L3 | >100 或 用尽 | 停止+报告 | - | ✅ 允许 |

### 神圣不可改变的规则

1. ✅ Git 备份点必须在开始前创建
2. ⛔ **禁止删除文件** — 包括 `DeleteFile` 工具、`Remove-Item`、`del` 等所有形式
3. ✅ 文件删除改为移动到 `.x/*.del`
4. ✅ **每个任务完成后必须立即 git 提交** — 无论任务大小
5. ✅ 每个里程碑必须立即 git 提交
6. ✅ 回滚后必须从第一个任务重新开始
7. ❌ 禁止用 PowerShell 修改 .cs 文件内容
8. ❌ 禁止跳过遗留检查直接执行
9. 🔴 **TDD 铁律**: 先写失败测试才允许写生产代码，禁止补测试
10. ✅ **E2E 覆盖**: 对外接口必须有 E2E 测试，E2E 变更必须人工逐行确认
11. ⚠️ **降级策略**: 错误过多时拆到单文件单函数级别，禁止整体回滚再重来
12. ⛔ **禁止中断任务**: 禁止因时间/长度关系中途停止，禁止主动询问"是否继续"
13. ✅ **穷尽手段**: Level 3 交互前必须穷尽 MCP记忆 → 项目代码 → 技能 → 互联网
14. ✅ **交付优先级**: 可运行 > 完美，先跑通核心路径再优化
15. ✅ **Git commit 决策记录**: 自主决策必须在 commit message 中说明
