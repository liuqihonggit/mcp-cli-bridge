# 架构迁移操作专项规则

> 📌 **适用场景**: 代码迁移、重构、命名空间修改等大规模代码变动
> ⚠️ **注意**: 本文件仅适用于迁移场景，不作为通用开发规则

## 🔴 子智能体使用限制

### 核心原则

**子智能体可能会重复你的迁移错误，因此使用场景必须严格限制**

### ✅ 允许使用的场景

1. **命名空间批量修改**
   - 使用 `powershell-batch-replace` 批量替换命名空间
   - 简单的字符串替换任务（如 `OldNamespace` → `NewNamespace`）
   - 不涉及逻辑变更的纯文本操作
2. **简单的重命名任务**
   - 类名/方法名批量重命名（不改变签名）
   - 文件名批量修改
   - using 语句的添加/删除
3. **格式化和清理**
   - 代码格式化
   - 删除空行、统一缩进
   - 添加标准文件头注释

### ❌ 禁止使用的场景

1. **复杂的迁移重构**
   - 跨项目的功能模块移动
   - 架构层面的调整（如从单体拆分为微服务）
   - 涉及多个文件联动的逻辑变更
2. **需要理解上下文的任务**
   - 接口实现类的迁移（需要理解业务逻辑）
   - 依赖注入配置的修改
   - 测试用例的同步更新
3. **容易出错的复杂操作**
   - 正则表达式匹配的复杂文本处理
   - 条件判断相关的代码生成
   - 需要编译验证的代码修改

### ⚠️ 为什么这样限制？

**问题根源**:

- 子智能体执行时你无法实时看到具体出错内容
- 一旦迁移错误，可能需要回滚大量文件
- 子智能体重复相同错误的概率较高
- 调试成本远高于手动执行

**实际案例**:

```
❌ 错误做法: 让子智能体迁移整个服务层
   结果: 50个文件被修改，其中30个有编译错误
   后果: 无法定位哪个文件出错，只能全部回滚

✅ 正确做法: 主智能体逐个文件迁移，每迁移一个就编译验证
   结果: 每次只改1-2个文件，立即发现并修复问题
   后果: 渐进式成功，可追溯每个步骤
```

## 📋 迁移操作检查清单

### 迁移前

- [ ] Git环境干净，已备份
- [ ] 明确迁移范围和目标
- [ ] 制定渐进式迁移计划（分步骤）
- [ ] 确认哪些可以用子智能体，哪些必须手动做

### 迁移中

- [ ] 每完成一个步骤立即编译
- [ ] 编译成功后提交一次git
- [ ] 主工程编译成功后，测试用例也要编译
- [ ] 发现错误立即停止，先修复再继续

### 迁移后

- [ ] 全量测试通过
- [ ] 无编译警告（或明确忽略的警告）
- [ ] 文档更新（如README、架构图）
- [ ] 记录迁移经验到记忆系统

## 💡 最佳实践

### 渐进式迁移示例

```markdown
## 目标: 将 AuthService 从 OldProject 迁移到 NewProject

### 步骤1: 移动文件（手动）
- 移动 AuthService.cs 到新位置
- 编译验证 ✅
- git commit

### 步骤2: 修改命名空间（可用子智能体）
- 批量替换 `using OldProject.Services` → `using NewProject.Services`
- 编译验证 ✅
- git commit

### 步骤3: 更新依赖注入（手动）
- 修改 Program.cs / Startup.cs 的注册
- 编译验证 ✅
- git commit

### 步骤4: 更新测试用例（手动）
- 修改引用和断言
- 运行测试 ✅
- git commit
```

### 子智能体正确使用示例

```powershell
# ✅ 正确: 批量替换命名空间（简单、可逆、易验证）
.\replace-namespace.ps1 -old "OldNamespace" -new "NewNamespace" -path "src/"

# ❌ 错误: 让子智能体"迁移整个认证模块"（复杂、不可控、难调试）
Task(subagent, "将AuthService迁移到新架构")  # 禁止！
```

## 🚨 常见错误

| 错误         | 原因     | 解决方案         |
| ---------- | ------ | ------------ |
| 一次性迁移50+文件 | 贪多求快   | 分成10批，每批5个文件 |
| 用子智能体改接口实现 | 上下文丢失  | 手动逐个处理       |
| 不编译就继续下一步  | 盲目乐观   | 每步必编译        |
| 忽略测试用例编译   | 只关注主工程 | 测试也必须通过      |
| 回滚后重复相同方案  | 未分析根因  | 换一种方式重试      |
