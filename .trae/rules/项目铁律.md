# 项目铁律
采取渐进式的方法,每次只完成一个功能,不要一次完成所有功能.
1,保证git环境干净,备份一次.
2,每次完成一个功能,都要编译成功之后提交一次git.
3,主工程编译成功了,测试用例也需要编译成功.
4,一旦有疑问,或者发现错误,立即停止,先修复,再继续.
5,当你无法完成时候,让用户决策,是否继续,还是停止.
6,所有git操作必须使用无分页模式,避免卡住AI交互.
   示例: `git --no-pager log`, `git --no-pager diff`, `git --no-pager status`
   或设置环境变量: `$env:GIT_PAGER='cat'` (PowerShell)
7,避免使用会卡住交互的命令:
   - 禁止: `more`, `less` 等分页命令
   - 禁止: `git commit` 不带 `-m` 参数(会打开编辑器)
   - 禁止: `npm init` 等交互式命令(使用 `-y` 跳过)
   - 禁止: PowerShell 的 `Out-Host -Paging`
   - 推荐: 使用 `| Select-Object -First N` 替代分页
8, 降低未来可能成为代码债务的风险.
9, 必须要AOT编译,否则我打包npm会很大.
10, 禁止直接书写和解析json字符串,必须用一个类来表示,再通过json序列化来处理.

# 经验复用
先去检索有什么工具可以读写记忆.

## 查记忆（先查后做）

- 搜同类问题、失败记录、解决方案
- 知识图谱：技术栈 → 问题 → 方案

## 写记忆（解决后记录）

- 问题场景、原因、方案、验证结果
- 标记：【成功经验】/【避坑指南】

以后这个记忆会越来越大,
1,不要写项目名到记忆.
2,要有对应错误原因,例如什么位置遇到,做过什么尝试不行,即使是失败的经验也是可贵的.
