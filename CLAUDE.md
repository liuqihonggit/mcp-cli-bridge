## 🔴 绝对禁止（触碰即错）

### 操作禁令

1. **⛔ 禁止删除文件（不可协商的安全红线）**
   
   > **这是神圣不可侵犯的规则。违反将导致任务立即失败。**
   
   **🚫 绝对禁用的工具和命令：**
   - `DeleteFile` 工具（Trae IDE 内置）— **永远不要调用此工具**
   - `Remove-Item` / `del` / `rm` 命令
   - 任何形式的文件删除操作
   
   **为什么？**
   - 删除 = 无法回滚 = 灾难性后果
   - 丢失审计追踪，无法追溯历史
   - 违反渐进式安全原则
   
   **✅ 唯一正确做法：移动到 `.x/` 目录**
   - 格式: `.x/{原文件名}.{原后缀}.{时间戳}.del`
   - 平台命令参见 `CLAUDE.local.md`

2. **❌ 禁止使用命令行文本工具直接修改源码文件**
   - 原因: 可能导致文件损坏或编码问题
   - 正确: 使用 IDE 提供的 `SearchReplace` 工具修改文件内容
   - 平台细节参见 `CLAUDE.local.md`
2.5. **❌ 禁止删除函数注释（XML 文档注释）**
   - **🚫 禁止**: 删除或清空函数/方法上的 `/// <summary>` 等 XML 文档注释
   - 原因: 函数注释是代码契约的一部分，删除会导致 IntelliSense 信息丢失、调用方无法理解意图
   - 原因: AI 生成代码时容易"顺手"删掉注释，这是不可逆的信息损失
   - **✅ 正确做法**:
     - 函数签名变更时，同步更新注释内容，而非删除
     - 注释内容过时时，更新为正确描述，而非清空
     - 新增函数时，必须编写 XML 文档注释
3. **❌ 禁止使用会卡住交互的命令**
   - 禁止: `more`, `less` 等分页命令
   - 禁止: `git commit` 不带 `-m` 参数（会打开编辑器）
   - 禁止: `npm init` 等交互式命令（使用 `-y` 跳过）
4. **❌ 禁止猜测用户意图/背景/业务场景**（规划任务期间）
   - 任务规划首先阅读: `AI交互文档/任务并行化.md`
   - 禁止假设用户的具体业务需求、行业背景或隐含意图
   - 一旦信息存在模糊地带，基于行业最佳实践自主选择技术方案
   - 优先选择保守、安全的实现方式
   - 记录决策依据到工作文件末尾（遵循第3条对话偏好）
   - 此时推荐使用 `AskUserQuestion` 工具，询问用户是否确认，或者建议其他实现方式
5. **⚠️ 分级交互控制**（执行任务期间）
   - **Level 1-2 禁止交互**: 自主决策技术方案，基于上下文选择最合理的实现方式
   - **Level 3 允许交互**: 错误超过100个或重试用尽时，使用 `AskUserQuestion` 工具请求用户决策
   - 📎 *详细分级标准见 [渐进式执行模式.md](AI交互文档/渐进式执行模式.md) > 阶段3*
6. **⛔ 禁止并行子智能体期间提交 Git**
   - 每次指派子智能体时，必须告知子智能体当前处于**并行期间**，禁止执行 `git commit` / `git push`
   - 原因: 并行子智能体操作同一仓库，提交会导致冲突或覆盖他人工作
   - ✅ 正确做法: 并行任务全部完成后，由主智能体统一提交
7. **⚠️ Git 提交前遇到错误时禁止强行提交**
   - 提交前若遇到错误（如文件锁定、合并冲突），可能是并行子智能体正在操作同一仓库
   - 此时**禁止**执行 `git commit`，应等待并行任务完成后再统一处理
   - 若持续失败，使用 `AskUserQuestion` 工具请求用户决策

## ✅ 必须执行（遗漏即错）

### 开发流程强制要求

1. **✅ 必须采用渐进式开发方法**
   - 每次只完成一个功能，不要一次完成所有功能
   - **每完成一个功能都要编译成功后提交git — 无论功能大小**
   - 即使是修改单个文件、重命名变量等小任务，完成后也必须立即提交
   - 禁止累积多个修改后统一提交
   - 主工程编译成功后，测试用例也需编译成功
   - 一旦有疑问或发现错误，**立即停止**，先修复再继续
   - 🔗 **例外**: `/spec` 模式下允许并行处理多个文件分区操作，详见 `AI交互文档/任务并行化.md`

1.5. **⚠️ 任务失败处理机制**
   - 📎 *详细流程见 [渐进式执行模式.md](AI交互文档/渐进式执行模式.md) > 阶段3*
   - **核心原则**: 必须先评估错误级别，禁止遇到小错误就立即回滚
   - **总上限**: 3次重试机会（含自行修复和回滚重试）
   - **⚠️ 降级策略**: 错误累积过多时，禁止反复重试整个任务触发git回滚
     - 正确做法: 将当前任务拆成更小的子步骤，每步独立编译验证
     - 拆到多小？拆到单文件单函数级别，确保每步可独立编译
     - 禁止: 看到大量错误就整体回滚再重来（会陷入回滚-重试死循环）

3. **🔴 TDD 铁律（测试驱动开发，强制）**
   - **循环**: 🔴红(写失败测试) → 🟢绿(写最小代码通过) → 🔵重构(优化)
   - **顺序**: 先写测试验证行为，后补测试验证输入
   - **铁律**: 必须先写一个**会失败**的测试，才允许写生产代码让它通过
   - **⛔ 禁止补测试**: 若发现生产代码没有对应测试，**禁止直接补测试**
     - 正确做法: 先移除该生产代码（移到 `.x/`），再从写失败测试开始
     - 原因: 补测试会依赖旧代码结构，破坏TDD驱动的正确设计
   - **例外**: 纯DTO/枚举等无逻辑代码无需TDD
   - **位置**: 写在对应的单元测试项目,而不是E2E.

3.5. **🔴 DTT 铁律（缺陷驱动测试，修复 bug 时强制）**
   - **循环**: 🐛发现bug → 🔴红(写失败测试复现bug) → 🟢绿(修复bug使测试通过) → 🔵重构(优化)
   - **铁律**: 遇到 bug 时，**禁止直接修改生产代码修复**
     - 正确做法: 先在单元测试中写一个**能复现该 bug 的失败测试**
     - 确认测试确实失败（证明 bug 被正确复现）
     - 然后修改生产代码修复 bug，使测试通过
   - **⛔ 禁止跳过复现步骤**: 直接改代码修 bug 而不先写测试 = 违规
     - 原因: 没有测试保障的修复，bug 可能回归且无法察觉
     - 原因: 复现过程帮助理解 bug 根因，避免治标不治本
   - **位置**: 写在对应的单元测试项目,而不是E2E

4. **✅ 必须有 E2E 测试覆盖协议边界**
   - 每个对外暴露的接口必须有E2E测试，模拟真实客户端和服务端交互
   - 目的: 防止参数顺序变更、协议格式破坏等隐性回归
   - **代码审查规则**: 只有E2E测试被修改时，才需要人工仔细审查变更内容
   - 单元测试变更可快速过审，E2E变更必须逐行确认

1. **✅ 必须保证Git环境干净后再开始工作**
   - 开始任务前先备份一次
   - 所有git操作必须使用无分页模式:
     - `git --no-pager log`
     - `git --no-pager diff`
     - `git --no-pager status`
   - git commit 消息必须使用中文
3. **🔴 Git 提交前必须通过完整验证流水线（强制）**
   - 每次 `git commit` 前，必须按顺序执行以下全部步骤，**全部通过后才允许提交**：
     1. **编译全部**: `dotnet build McpHost.slnx -c Release` — 确保零错误零警告
     2. **单元测试**: `dotnet test McpHost.slnx -c Release` — 确保全部通过
     3. **E2E测试**: 运行E2E测试项目 — 确保协议边界无回归
     4. **发布全部**: 若项目根目录存在 `build.ps1`，必须执行 `.\build.ps1` — 确保发布产物正确
     5. **修复全部**: 上述任何步骤失败，必须修复后再重新从第1步开始，禁止跳过任何步骤提交
   - **⛔ 禁止跳过步骤**: 即使只改了一个注释，也必须走完整个流水线
   - **⛔ 禁止部分通过就提交**: 单元测试通过但E2E失败 = 不允许提交
   - **原因**: 每次提交都应是可发布状态，避免积累隐性缺陷
2. **✅ 必须支持 NativeAOT 兼容编译**
   - 所有代码必须兼容 NativeAOT 编译
   - 禁止使用动态类型、反射 emit、直接解析 JSON
   - 📎 *详细要求和发布约束见 AGENTS.md > 关键约束*
   - 📎 *具体编码约束见 [代码规范.md](AI交互文档/代码规范.md)*

## 🔄 强制性工作流程

### 经验复用机制（先查后做）

1. **查记忆（开始任务前必做）**
   - 搜同类问题、失败记录、解决方案
   - 知识图谱: 技术栈 → 问题 → 方案
   - 不要重复造轮子，避免重蹈覆辙
2. **写记忆（解决问题后必做）**
   - 记录: 问题场景、原因、方案、验证结果
   - 标记: 【成功经验】 / 【避坑指南】
   - 要有对应错误原因（什么位置遇到，做过什么尝试不行）
   - 即使失败的经验也是可贵的
3. **注意事项**
   - 不要写项目名到记忆（记忆会越来越大，要保持通用性）
   - 先去检索有什么工具可以读写记忆

### 渐进式迁移策略（重构时必用）

> 📎 *详细流程见 [渐进式执行模式.md](AI交互文档/渐进式执行模式.md) > 阶段2*

1. 保证git环境干净，备份一次
2. 每次移动一个功能模块
3. 移动后立即编译验证
4. 编译成功后提交git
5. **禁止一次性大规模重构**

### 对话偏好补充

1. **涉及文件更改时要先列目录树**
2. **架构不合理要提出来，不要直接生成代码**
3. **✅ 渐进式成功后必须记录自主决策**
   - **时机**: 每完成一个功能点并编译成功后，立即记录
   - **位置**: 写到当前工作文件的末尾（不是CLAUDE.md）
   - **格式**: 使用 `<!-- 🤖 Auto Decision: [决策内容] -->` 注释格式
   - **内容**: 说明做了什么决策、为什么这样选择、替代方案是什么
   - **示例**:
     ```markdown
     <!-- 🤖 Auto Decision: 2026-04-30 -->
     <!-- 决策: 使用 FrozenDictionary 替代 switch-case -->
     <!-- 原因: 性能更优，符合NativeAOT要求，避免硬编码 -->
     <!-- 替代方案: 特性标记（复杂度较高，暂不采用）-->
     <!-- 验证: 编译通过，测试用例全部通过 ✅ -->
     ```
   - **⚠️ 重要**: 未编译成功的决策不得记录，必须先修复错误
4. **结束对话时有未完成的工作或缺陷，一定要⚠️emoji表情提醒**
